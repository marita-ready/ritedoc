# RiteDoc Data Handling Policy

This Data Handling Policy details the strict procedures and technical safeguards implemented by ReadyCompliant to ensure the absolute privacy and security of data processed by the RiteDoc software application.

## 1. Core Principle: Offline-Only Processing

The fundamental architecture of RiteDoc is built on the principle of offline-only processing. The software is designed to operate entirely on the Client's local machine without transmitting any participant data, progress notes, or personally identifiable information (PII) to any cloud server, external database, or third-party service.

## 2. PII Scrubbing Process

Before any progress note is processed by the local AI compliance engine, it must pass through the proprietary Nanoclaw PII Scrubber, written in Rust.

### 2.1 Identification and Redaction

The PII Scrubber scans the raw CSV input and identifies potential PII, including but not limited to:

*   Participant names (first, last, and known aliases)
*   Support worker names
*   Associate names (family members, kinship titles like "Aunty", "Uncle", "Nan", "Pop")
*   Medical professionals and facility names
*   Locations (addresses, specific facility names, Victorian suburbs)
*   Contact details (phone numbers, email addresses)

### 2.2 Replacement Tags

Identified PII is immediately redacted and replaced with standardised, non-identifying tags (e.g., `[Participant]`, `[Support Worker]`, `[Associate]`, `[Location]`, `[Phone]`, `[Email]`, `[Address]`).

This scrubbing process occurs entirely in the local memory of the Client's machine. The original, unscrubbed data is never transmitted or stored outside the local application environment.

## 3. Local Compliance Engine

Following the PII scrubbing process, the redacted notes are analysed by the local AI compliance engine (Nanoclaw/llama.cpp running the Phi-4-mini Q4_K_M model).

### 3.1 Standard vs. Turbo Mode

The software automatically detects the hardware capabilities of the Client's machine to determine the processing mode:

*   **Standard Mode:** Utilises 2 agents (Agent 1: Rewrite + Red Flag Scan, Agent 2: Audit + Score) for machines without a dedicated GPU (e.g., 16GB RAM, i7 processor).
*   **Turbo Mode:** Utilises 3 agents (Agent 1: Rewrite, Agent 2: Red Flag Scan, Agent 3: Audit + Score) for machines with a dedicated GPU or higher specifications.

Regardless of the mode, all AI inference and processing occur locally. The model weights and parameters are stored locally and are not updated or modified based on the Client's data.

## 4. 48-Hour Local Auto-Delete Policy

To further mitigate data security risks, RiteDoc enforces a strict 48-hour local auto-delete policy.

### 4.1 Data Deletion

All data associated with a processing session, including:

*   Imported raw CSV files
*   Temporary scrubbed data files
*   Processed audit-prepared drafts
*   Internal compliance scores and traffic light indicators

are permanently and irretrievably deleted from the local application storage exactly 48 hours after the processing session concludes. This ensures that sensitive data does not accumulate on the Client's machine over time.

## 5. Software and Cartridge Updates

ReadyCompliant periodically releases updates to the RiteDoc software and the state-specific compliance cartridges (JSON files containing red flags, rubrics, policies, and system prompts).

### 5.1 Update Mechanism

These updates are delivered via a secure, encrypted connection. The update process involves downloading the new software version or cartridge file to the Client's machine.

### 5.2 No Data Transmission During Updates

Crucially, the update process is strictly one-way (download only). No participant data, progress notes, or processing history is ever transmitted back to ReadyCompliant during a software or cartridge update. The updates solely modify the internal logic and reference materials used by the local compliance engine.

## 6. Minimal Telemetry and Support Data

The only data transmitted from the RiteDoc application to ReadyCompliant consists of minimal telemetry and support-related information:

*   **Licence Validation:** Periodic checks to verify the validity of the licence key.
*   **Hardware Telemetry:** Basic hardware specifications (RAM, CPU, GPU presence) to determine the appropriate processing mode (Standard vs. Turbo).
*   **Crash Reports:** Anonymous crash data captured by Sentry to assist in identifying and resolving software bugs. This data explicitly excludes any participant PII or progress note content.
*   **Support Tickets:** Information provided by the Client when submitting a support ticket, including diagnostic logs generated by the self-fix system or Dify bot.

This data is transmitted securely and stored in our Supabase database in accordance with our Privacy Policy.
